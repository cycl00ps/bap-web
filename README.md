# BAP Web

`bap-web` is a Go + HTMX control plane for managing BAP Firecracker microVMs.

The implementation is API-first:

- `/api/...` returns JSON for automation.
- `/ws/...` handles logs, events, and terminal streams.
- `/ui/...` returns HTMX fragments.
- HTML pages are a human client of the same service layer.
- `/openapi.json`, `/docs/api`, `/docs/agents.md`, and `/llms.txt` expose API and agent-facing documentation.

## Development

```bash
go mod download
go test ./...
go run ./cmd/bap-webd --config ./config.example.yaml
```

Host integration tests and real VM launch require root privileges, KVM, Firecracker, jailer, nftables, and a base rootfs.

Run the live VM smoke against a running `bap-webd` service with:

```bash
BAP_WEB_RUN_LIVE_VM_SMOKE=1 ./scripts/dev-integration.sh
```

## Installed Service

The development host can run `bap-webd` as a root-owned systemd service:

```bash
sudo bash ./scripts/install-systemd.sh
sudo systemctl status bap-webd --no-pager
```

Default local URL:

```text
http://127.0.0.1:8080
```

Initial admin credentials are generated by:

```bash
sudo bash ./scripts/bootstrap-admin.sh
```

The generated credentials are stored root-only at:

```text
/etc/bap-web/initial-admin.txt
```

## Host Prep Scripts

```bash
sudo bash ./scripts/host-check.sh
```

## Current V1 Capabilities

- Local admin setup/login with secure session cookies and CSRF checks.
- API-first VM management under `/api/...`.
- HTMX-rendered dashboard and VM detail pages.
- SQLite registry at `/var/lib/bap-web/bap-web.db`.
- EC2-style SSH key registry under `/api/ssh-keys`; generated private keys are returned once and are not stored.
- Browser terminal access uses short-lived in-memory SSH keys published to guests through the VM metadata key endpoint.
- Base rootfs image registry under `/api/base-images`, with default registration from `paths.base_rootfs`.
- Admin-managed base image build jobs and build hooks under `/api/base-images/build` and `/api/image-hooks`.
- VM creation can select a base image and requested derived rootfs size; sparse/reflink copies are used when the filesystem supports them.
- Kernel registry under `/api/kernels`, with default registration from `paths.kernel_image`.
- Admin kernel management UI for Firecracker CI imports, local uploads, status/boot-arg edits, deletion, and test-job log polling.
- VM creation can select an active tested kernel; kernel test jobs boot an ephemeral VM and mark successful kernels active.
- VM create/start/stop/delete.
- VM restart, stopped resource edits, and restart-with-new CPU/memory values.
- VM detail management sections for overview, shell, networking, resources, and logs.
- Public OpenAPI, Swagger UI, and LLM/agent instruction pages.
- Host status checks in the management UI.
- Firecracker + jailer launch.
- Registry-backed `/30` routed point-to-point allocation.
- Optional registry-backed shared bridge networks for multiple microVMs on one subnet.
- Registry-backed custom ingress rules mapping host TCP/UDP ports to guest ports.
- Reusable egress policies with allow-all, deny-all, and restricted TCP/UDP/CIDR modes.
- nftables ingress, egress, NAT, anti-spoofing, and protected-host rules.
- Runtime firewalld metadata rules for guest access to the key endpoint.

## Smoke-Tested Host State

The current host has a running smoke VM:

```text
name: smokevm
guest_ip: 172.31.1.2
ssh_port: 20000
state: running
```

External SSH uses the private key exported when the selected SSH key was generated:

```bash
ssh -i <private-key-file> dev@<host-lan-ip> -p 20000
```

LAN clients should connect to the host LAN IP on the mapped SSH port.
